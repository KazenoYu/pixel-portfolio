每次討論前端技術選型，SSR vs CSR 都會被搬上桌。網路上資訊很多但常常被框架名稱（Next.js / Nuxt / Vite）綁架——你選的不是技術而是 brand。

這篇用兩個 yes / no 問題把選擇拆乾淨。

## 兩種渲染的本質差異

### CSR — 客戶端渲染

瀏覽器下載 JavaScript，**在瀏覽器上執行**，畫面才出現。

```
瀏覽器下載 → 空白頁面 → 下載 JS → 執行 JS → 畫面出現
                ↑
        使用者看到空白（約 1-2 秒）
```

代表工具：React + Vite、Vue + Vite。
特點：首次載入稍慢，但之後操作流暢、不需要伺服器持續運算。

### SSR — 伺服器端渲染

伺服器**先把完整 HTML 產生好**，瀏覽器直接顯示。

```
伺服器產生完整 HTML → 瀏覽器下載 → 畫面立刻出現 → JS 載入後可互動
                                        ↑
                                  幾乎馬上看到內容
```

代表工具：Next.js、Nuxt、SvelteKit。
特點：首次載入快、SEO 友善，但需要伺服器持續運算。

## 用 SSR 的場景

| 情境 | 為什麼需要 SSR |
|---|---|
| 電商前台（商品頁） | 商品要被 Google 搜尋到 → 需要 SEO |
| 部落格 / 內容網站 | 文章要被 Google 收錄 → 需要 SEO |
| 行銷官網 | 要讓潛在客戶搜尋到 → 需要 SEO |
| 社群媒體分享 | 連結要有預覽圖 → 需要 SSR 生成 meta tags |
| C 端消費者頁面 | 公開頁面，必須能被搜尋到 |

## 用 CSR 的場景

| 情境 | 為什麼不需要 SSR |
|---|---|
| POS 收銀系統 | 內部系統，不需要 SEO；需離線運作，CSR 更合適 |
| ERP 後台管理 | 需登入才能用，Google 不需要看到 |
| Dashboard 儀表板 | 內部使用，資料即時更新比 SEO 重要 |
| 任何登入牆後的系統 | 登入牆後的內容不需要被搜尋引擎收錄 |

## 兩個問題決定方向

> 問自己：
>
> 1. 這個頁面需要被 **Google 搜到**嗎？
> 2. 這個系統需要**離線運作**嗎？

| Q1 搜尋 | Q2 離線 | 走哪 |
|---|---|---|
| 要 | 不要 | **SSR**（Next / Nuxt） |
| 不要 | 要 | **CSR**（Vite + React/Vue） |
| 不要 | 不要 | 兩者都可以，看你舒服哪個 |
| 要 | 要 | 罕見組合；通常拆兩個 app |

絕大多數情境兩個問題就分得開。剩下的邊界情況用 monorepo 處理。

## Monorepo 允許兩者並存

在做 POS / ERP 系統時，內部後台需要離線（CSR），如果未來要加一個面向消費者的線上點餐頁面（C 端，需 SEO），完全可以在同一個 monorepo 裡新增一個 Next.js app：

```
apps/
├── web/           # React + Vite（ERP 後台）       ← CSR
├── pos-terminal/  # React + Vite（POS 收銀）       ← CSR
└── storefront/    # Next.js（線上點餐 / 電商前台）  ← SSR（C 端需要時再加）
```

不同的 app **可以用不同的技術，但共享同一套 types 和邏輯**。這就是 monorepo 真正的價值——不用為了「全站統一」而把錯的技術硬套到不適合的地方。

## 一個實例的決定理由

我在做的 POS / ERP 系統選擇 **React + Vite（CSR）**，理由：

- **不需要 SEO**：員工登入才能用，搜尋引擎不需要看到
- **需要離線能力**：收銀員不能因為網路斷線就無法結帳——CSR 更好支援 service worker / offline-first
- **架構職責分離**：NestJS 後端專注 API，React 前端專注 UI，前後端分離清楚
- **build 簡單**：純靜態檔，CDN 部署、不用維護 Node 伺服器

如果以後要做 C 端線上點餐，就再加一個 Next.js storefront。**讓對的工具做對的事**，不是強迫整站統一。

## 結語

技術選型容易被「最近什麼框架紅」綁架，但回到第一性原理：**這個頁面服務的對象是誰、Google 要不要進得來、需不需要離線**——這三件事決定一切，框架名字只是工具。

下次再聽到「不用 Next.js 就落伍了」這種話，先反問一句：「你的頁面要被 Google 搜到嗎？」
